5. Mini Git 구축
분야: AI/SW 기초
구분: 자료구조와 알고리즘
학습시간: 80시간

미션:
Git의 핵심 구조인 커밋 그래프, 브랜치, 탐색, 검색, 정렬을 CLI 기반 Mini Git으로 구현한다.

최종 결과물:
- CLI 기반 Mini Git 프로그램
- main.py
- README.md

핵심 명령:
- INIT <user_name>
- BRANCH <branch_name>
- SWITCH <branch_name>
- COMMIT <message>
- LOG
- PATH <commit1> <commit2>
- ANCESTORS <commit_hash>
- SEARCH <keyword>
- SEARCH --author=<name>
- LOG --sort-by=date|author
- exit 또는 quit

핵심 요구사항:
- 명령어 대소문자 구분 없음
- 공백 포함 문자열은 따옴표 허용
- 잘못된 입력 에러 메시지 표준화
- 커밋 노드 필드: hash, message, author, timestamp, parents
- 커밋 그래프는 DAG
- hash로 빠르게 커밋 조회
- 세션 내 hash 중복 금지
- 역색인 구현: keyword -> commit_hash 목록, author -> commit_hash 목록
- Python sorted(), list.sort() 사용 금지
- 직접 정렬 알고리즘 구현
- LOG는 부모 커밋이 자식보다 먼저 출력되도록 구현
- PATH는 무방향 간선 기준 최단 경로
- 최단 경로가 여러 개면 hash 경로 문자열 사전순 가장 작은 것 선택
- ANCESTORS는 모든 조상 출력

보너스:
- diff <file1> <file2>
- merge <branch_name>로 부모 2개인 merge commit 생성
- 정렬 알고리즘 성능 비교

개발 환경:
- Python 3.10 이상
- 그래프 전용 라이브러리 금지
- 정렬 표준 API 금지

6. Mini Redis 구축
분야: AI/SW 기초
구분: 자료구조와 알고리즘
학습시간: 80시간

미션:
해시맵, 이중 연결 리스트, 힙을 직접 구현하여 Redis 스타일 In-Memory Key-Value 저장소를 만든다.

최종 결과물:
- CLI 기반 Mini Redis 프로그램
- String 타입 기본 명령어
- 메모리 관리
- TTL 관리
- REPL

핵심 명령:
- SET key value
- GET key
- DEL key
- EXISTS key
- DBSIZE
- KEYS
- CONFIG SET maxmemory bytes
- INFO memory
- EXPIRE key seconds
- TTL key
- exit 또는 quit

핵심 요구사항:
- 이중 연결 리스트 직접 구현: insert_front, insert_back, remove_front, remove_back, remove_node, move_to_front
- 해시맵 직접 구현: put, get, remove, contains, keys, size
- 체이닝 방식 충돌 해결
- 로드 팩터 0.75 초과 시 버킷 2배 확장
- 최소 힙 직접 구현: push, pop, peek, size, heapify_up/down
- Redis 스타일 출력: OK, (nil), (integer) N, (error)
- 만료 여부를 먼저 확인
- SET 성공 시 LRU 추적 업데이트
- 기존 키 덮어쓰기 시 TTL 초기화
- GET 성공 시 LRU 갱신
- DEL은 데이터/TTL/LRU에서 함께 삭제
- used_memory = Σ(len(utf8(key)) + len(utf8(value)))
- maxmemory 초과 시 LRU부터 제거
- 단일 엔트리가 maxmemory 초과 시 저장하지 않고 OOM 출력
- EXPIRE 없는 키는 (integer) 0
- TTL 없는 키는 (integer) -1, 없는 키는 (integer) -2

보너스:
- 동적 배열 직접 구현
- STACK_QUEUE_DEQUE.md 문서화
- 이진 트리와 순회
- BST
- Pub/Sub

개발 환경:
- Python 3.8 이상
- dict, set, collections 사용 금지
- 네트워크 통신과 영속성 없음

8. 리눅스 프로세스 및 시스템 리소스 트러블슈팅
분야: AI/SW 기초
구분: Linux와 OS
학습시간: 40시간

미션:
제공된 agent-leak-app을 실행하고 OOM Crash, CPU Latency, Deadlock 장애를 관제 로그와 시스템 도구로 분석하여 GitHub Issue 형태 보고서를 작성한다.

최종 결과물:
- OOM Crash 이슈 리포트
- CPU Latency 이슈 리포트
- Deadlock 이슈 리포트
- 각 리포트는 PDF 또는 GitHub Repository 링크로 제출

리포트 필수 구조:
- Description: 현상 설명
- Evidence & Logs: monitor.sh 로그, 프로그램 로그, ps/top 등 증거
- Root Cause Analysis: 원인 분석
- Workaround & Verification: 환경변수 조정과 Before/After 비교

사전 준비 조건:
- root가 아닌 일반 사용자로 실행
- AGENT_HOME 필수
- AGENT_PORT=15034
- AGENT_UPLOAD_DIR=$AGENT_HOME/upload_files
- AGENT_KEY_PATH=$AGENT_HOME/api_keys
- AGENT_LOG_DIR 존재 및 쓰기 권한
- MEMORY_LIMIT 정수 50~512 MB
- CPU_MAX_OCCUPY 정수 10~100 %
- MULTI_THREAD_ENABLE true/false
- $AGENT_HOME/api_keys/secret.key 내용은 agent_api_key_test
- 0.0.0.0:15034 바인딩 가능

케이스별 최소 증거:
- OOM: 메모리 상승 수치, 종료 로그, MEMORY_LIMIT 변경 전후 비교
- CPU: CPU 급상승 캡처, WATCHDOG/SIGTERM 로그, CPU_MAX_OCCUPY 변경 전후 비교
- Deadlock: PID 존재, CPU/MEM 정체, WAITING/BLOCKED 마지막 로그, 스레드/락 대기 추론

보너스:
- 로그 타임스탬프로 스케줄링 알고리즘 추론

개발 환경:
- 제공 바이너리 실행 가능한 리눅스 환경
- 로컬 또는 Docker/VM 권장
- 디컴파일과 리버스 엔지니어링 금지

- monitor.sh 소스코드

수행 내역서 필수 증거:
- SSH 포트 20022 변경 및 Root 원격 접속 차단 확인
- UFW 또는 firewalld 활성화 및 20022/tcp, 15034/tcp만 허용
- agent-admin, agent-dev, agent-test 계정 생성
- agent-common, agent-core 그룹 생성
- 디렉토리 구조와 권한/ACL 확인
- 앱 Boot Sequence 5단계 OK 및 Agent READY 확인
- monitor.sh 실행 결과
- /var/log/agent-app/monitor.log 누적 기록
- agent-admin crontab 매분 실행 등록 및 1분 후 로그 증가 확인

핵심 요구사항:
- SSH 포트 20022
- PermitRootLogin 차단
- 방화벽 인바운드는 TCP 20022, TCP 15034만 허용
- agent-common: admin, dev, test
- agent-core: admin, dev
- AGENT_HOME, upload_files, api_keys, /var/log/agent-app 구성
- upload_files는 agent-common 그룹 R/W
- api_keys와 /var/log/agent-app은 agent-core 전용 R/W
- AGENT_HOME 예: /home/agent-admin/agent-app
- AGENT_PORT=15034
- AGENT_UPLOAD_DIR=$AGENT_HOME/upload_files
- AGENT_KEY_PATH=$AGENT_HOME/api_keys/t_secret.key
- AGENT_LOG_DIR=/var/log/agent-app
- 키 파일 내용: agent_api_key_test
- 루트 실행 금지
- 앱은 0.0.0.0:15034 LISTEN

monitor.sh 요구사항:
- 위치: $AGENT_HOME/bin/monitor.sh
- 소유자: agent-dev
- 그룹: agent-core
- 권한: 750
- cron 실행 계정: agent-admin
- 프로세스 실행 상태 확인, 비정상 시 exit 1
- TCP 15034 LISTEN 확인, 비정상 시 exit 1
- 방화벽 비활성 시 WARNING 출력 후 계속
- CPU, MEM, Root disk 사용률 수집
- CPU > 20%, MEM > 10%, DISK_USED > 80% WARNING
- 로그 파일: /var/log/agent-app/monitor.log
- 로그 포맷: [YYYY-MM-DD HH:MM:SS] PID:... CPU:..% MEM:..% DISK_USED:..%
- monitor.log는 최대 10MB/10개 파일 유지
- crontab으로 매분 실행

보너스:
- report.sh로 평균/최대/최소 요약
- 7일 경과 로그 압축, 30일 경과 아카이브 삭제

개발 환경:
- Ubuntu 22.04 LTS 또는 동등 리눅스 환경
- Bash만 사용
